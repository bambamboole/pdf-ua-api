pdf-ua-api focuses on accessible, validated PDF generation. Gotenberg is a
general Docker conversion API around Chromium, LibreOffice, and PDF engines.
WeasyPrint is a Python/CLI renderer built around HTML, CSS, and paged-media
document layout.

## Feature matrix

Legend: `✓` means first-class support, `✕` means not the intended product
surface, and `*` means supported with important caveats.

| Capability | pdf-ua-api | Gotenberg | WeasyPrint |
| --- | --- | --- | --- |
| Stateless Docker HTTP API | ✓ Stateless Ktor container with optional auth, rate limiting, OpenTelemetry, and upload integration. | ✓ Stateless Docker HTTP API; conversion requests are multipart and return generated files. | ✕ Library/CLI by default; you need to wrap it to expose an HTTP service. |
| HTML input | ✓ `/convert` accepts HTML directly. | ✓ Chromium HTML route accepts an uploaded `index.html`. | ✓ Accepts HTML strings, files, URLs, and file-like objects. |
| URL input | ✓ `/render/url` fetches remote HTML with SSRF protections. | ✓ Chromium URL route renders a remote page. | * URL input is available, but production services usually need a custom URL fetcher/policy. |
| JSON template model | ✓ First-class template schema, blocks, rows, attachments, docs, and embeddable builder tooling. | ✕ No native structured template schema. | ✕ No native service template schema. |
| Office and non-HTML formats | ✕ Not a goal; use HTML, URL, or template JSON. | ✓ LibreOffice routes handle Microsoft Office, OpenDocument, plain text, and more. | ✕ Not a goal; use HTML/CSS as source. |
| PDF/UA output | ✓ Core purpose; benchmark outputs pass PDF/UA-1. | * PDF engine exposes `pdfua`, but Chromium benchmark output did not pass PDF/UA. | * Supports PDF/UA variants, but valid output depends on source HTML/CSS and enabled PDF features. |
| PDF/A output | ✓ Default production output is PDF/A-3a + PDF/UA. | * PDF/A options exist, with metadata/post-processing caveats. | ✓ Supports PDF/A variants via `--pdf-variant` / `pdf_variant`. |
| Built-in validation | ✓ veraPDF validation endpoint for PDF/A-3a and PDF/UA-1. | ✕ Conversion-focused; validation is not the central API surface. | ✕ Generate first; validate with external tooling. |
| Browser fidelity and JavaScript | ✕ OpenHTMLToPDF is deterministic print-style rendering, not a browser runtime. | ✓ Chromium rendering, JavaScript handling, cookies, headers, waits, and network controls. | ✕ Print CSS renderer, not a JavaScript/browser runtime. |
| Headers, footers, and page layout | ✓ Template page/footer config plus HTML/CSS support. | * Chromium supports `header.html` and `footer.html`, with separate-context limitations. | ✓ Strong CSS paged-media support with `@page`. |
| Attachments and metadata | ✓ Template attachments, producer metadata, and document IDs. | ✓ PDF engines support metadata and embedded files; extra PDF tools include merge, split, watermark, and stamp. | ✓ Attachments, XMP metadata, custom metadata, sRGB, PDF forms, and image optimization options. |
| Remote asset security controls | ✓ Blocks private/loopback/link-local destinations by default; supports allow lists, timeouts, and size limits. | ✓ Outbound URL filtering and module-level network configuration. | * URL-fetch controls exist, but service-level policy is your responsibility. |

## What the numbers say

- **Throughput.** pdf-ua-api is fastest across the small, medium, and mixed
  large accessible-output runs in this result set. Gotenberg Chromium is included
  as a non-PDF/UA rendering baseline and remains useful for raw browser-output
  comparison.
- **Accessibility.** Every pdf-ua-api output passes PDF/UA-1 validation. With
  `embedColorProfile=true`, the normal API output also passes PDF/A-3a.
  WeasyPrint's `pdf/ua-1` output passes PDF/UA-1 on the simplest
  document but is flagged as non-compliant on the more complex ones. Gotenberg
  Chromium output is included as a rendering baseline, not as a PDF/UA engine.
- **File size.** pdf-ua-api produces larger files — but not because WeasyPrint
  skips font embedding. The engines embed the same subsetted *Liberation Sans*.
  The normal pdf-ua-api default packs an ICC color profile, a PDF/A output intent,
  **and** a full accessibility tag tree into one file. The benchmark disables the
  color profile for pdf-ua-api so the size column compares accessible output mode
  against WeasyPrint's `pdf/ua-1` mode while showing where Chromium-only output
  lands. Re-run with
  `PDF_UA_EMBED_COLOR_PROFILE=true` to measure the combined archival accessible
  output.

## Methodology

All engines run as HTTP services in Docker and receive the exact same HTML per
document. Gotenberg receives it as a multipart `index.html` upload with
`preferCssPageSize=true` and `printBackground=true`. A warmup phase precedes
measurement so cold-start does not skew results — disclosed, not hidden.
pdf-ua-api's rate limiter is disabled for the run so the load test measures raw
engine throughput.

The corpus has three document shapes: a small text document, a medium invoice
with tables, and a large mixed report with paragraphs, figures, callouts, and a
limited number of tables. The large fixture intentionally avoids being a pure
table stress test.

The containers ship Liberation Sans, so the engines embed the same typeface and
file size isn't skewed by font choice. Throughput and latency come from
[`oha`](https://github.com/hatoo/oha). Every output PDF — from **all** engines —
is validated by pdf-ua-api's own veraPDF `/validate` endpoint, so all are judged
by the same validator. The default run disables pdf-ua-api's embedded color
profile for the benchmark request only; the API's normal default remains PDF/A-3a
and PDF/UA-1 output. File size is the output PDF byte length.

Numbers reflect a single machine and a single run. Re-run them yourself.
