## What the numbers say

- **Throughput.** pdf-ua-api sustains an order of magnitude more requests per
  second at a fraction of the latency across every document tier.
- **Accessibility.** Every pdf-ua-api output passes both PDF/A-3a and PDF/UA-1
  validation. WeasyPrint's `pdf/ua-1` output passes PDF/UA-1 on the simplest
  document but is flagged as non-compliant on the more complex ones.
- **File size.** pdf-ua-api produces larger files. This is the cost of PDF/A-3a:
  fonts are fully embedded so the document renders identically anywhere, forever.
  It is a deliberate trade-off for archival accessibility, not an oversight.

## Methodology

Both engines run as single-worker HTTP services in Docker and receive the exact
same HTML per document. A warmup phase precedes measurement so JVM cold-start
does not skew results — disclosed, not hidden. pdf-ua-api's rate limiter is
disabled for the run so the load test measures raw engine throughput (WeasyPrint
has no limiter, so both run unthrottled).

Throughput and latency come from [`oha`](https://github.com/hatoo/oha). Every
output PDF — from **both** engines — is validated by pdf-ua-api's own veraPDF
`/validate` endpoint against PDF/A-3a + PDF/UA-1, so both are judged by the same
validator. File size is the output PDF byte length.

Numbers reflect a single machine and a single run. Re-run them yourself.
